<b>Explaination: </b>

i) Primary Clustering: No

In this particular probing sequence for an open address hash, the probing methodology does not probe contiguous locations in the the has table T. Primary clustering is an inherent characteristic of linear probing sequencing, and this hash function is not linear nor exhibits contiguous probing. 

ii) Secondary Clustering: Yes

With respect to the probing sequence, clusters will begin to form in locations relativistic to the sequence and not contiguously. Essentially probing will occur with broadening jump gaps as the sequence computes. This such behavior will result in initially small clusters but as i gets very large so will these clusters. Clusters will form with respect to the sequence which is bad for a few reasons. Secondary clustering will result in collisions -- especially when these separate locations of clusters become very large.

iii) Always able to add a key if the table is not full: Yes

This such open address probe sequence will result in being able to add a key to a location in the table if it not full. Unfortunately depending on the size of m for the hash table T, this process may take a long time, which could make the sequence inefficient. This such probing sequence also could result in many collisions assuming there is table wrap-around (sequence  % m). ---++ Problem 2 - Selection 

---+++ i. Median of two
---++++ Algorithm
<b>Explaination: </b>

Given X0 and X1 are both sorted arrays of length n. I will devise a algorithm of comparing medians.

X0[1~n] has its median at X0[n/2] lets label this m1

X1[1~n] has its median at X1[n/2] lets label this m2

Now we have 3 potential cases:

m1 = m2, m1 < m2, m1 > m2

case1
m1 = m2:
median = m1 = m2

case2
m1 < m2:

cut X0 in half X0[n/2] to X0[n]
cut X1 in half X1[0] to X1[n/2]

These halves will indefinitely contain the median of the merged array, so repeat the initial algorithm recursively.

case3
m1 > m2:

cut X0 in half X0[0] to X0[n/2]
cut X1 in half X1[n/2] to X1[n]

Again these halves will contain the median of the merged array.

If the case is case 2 or 3 recursively execute this algorithm until the arrays are made the same size. The result will yield the median of both arrays.

---++++ Analysis and running time
<b>Explaination: </b>

The best case run time is when the two arrays have the same median, and in that case the run time would be O(1). In worst case this algorithm will have to call recursively which will yield a run time of O(log(n)) in order to compute the median.
---+++ ii. Median of three
---++++ Algorithm
<b>Explaination: </b>

Given three sorted arrays of size n, in constant time retrieve the medians of all the sorted arrays. For each array they will be stored at X_k[n/2] (k is the number of arrays - 1). The next step is to find the min and max median. Lets call the min median 'a' and the max median 'b'. Explicitly stated: X_max[n/2] = b, X_min[n/2] = a. Then throw out  X_min[0]  to  X_min[n/2], and then also throw out  X_max[n/2] to  X_max[n]. The result of this will be that with each iteration the information outside of the bounds of the median will be discarded in constant time. This function will need to be recursively called so when discarding information it best in practice to discard the same amount of elements from both  X_min and  X_max. Since over time the arrays will become different lengths be sure to discard the smaller of the two boundary distances in  X_max and  X_min.

Continue this process recursively. Eventually there will only be two arrays left where you will then have your median of the three arrays. This methodology runs off of the assumptions that finding the median in a given array is constant time and discarding information outside of the  X_min and X_max bounds is also constant time. This approach is also a slightly extrapolated version of the approach illustrated in part i of this homework question. 

At the end of the algorithm there will only be one or two arrays left. a and b then must be considered exclusively. Only erase up to a and down to b exclusively. Continue this process and the median will present itself.
---++++ Analysis and running time
<b>Explaination: </b>

The run time of this algorithm is very efficient. This median finding algorithm runs at O(log(n)), which is extremely fast. The best case would be O(1) given the three arrays have the same median, but that is highly unlikely, so it is safe to say that the average case is O(log(n)) run time. This also of course depends on the behavior of the median function, time to de-allocate arrays, and iteration speed -- all of which are assumed to be executed in constant time for this algorithm.
---+++ iii. Median of n

 
---++++ Algorithm
<b>Explaination: </b>

The previous question regarding three sorted arrays of size n's approach can be extrapolated to solving this problem. Solve for all medians of all n sorted arrays of size n. Then find the min of the medians and the max of the medians. Labeled 'a' the min and labeled 'b' the max of the medians. The next step would be to look at the arrays a and b are found it. b constitutes and upper bound and a constitutes a lower bound. In the array containing a, discard all values less than or equal to a and in the the array containing b, discard all values greater than or equal to b. Inclusively considering a and b.

Now you know that all values between a and b when considering all n arrays must contain the median of all the arrays. Continue the process outlined above recursively until one or two arrays are left. Once this condition is met execute the process outlined in part 1 of this question. The result will yield the median.
---++++ Analysis and running time
<b>Explaination: </b>

All together this entire process will compute the median in O(nlog(n)) which is highly efficient and quick. This algorithm would operate under the same conditions of part 2's run time and time complexity. The best case but extremely unrealistic case would be O(1) which is the case were all n sorted arrays of size n all had the same exact median value, fortunately the average run time of the algorithm is O(nlog(n)). It not longer is only log(n) because there are now n arrays.

    string result;

myBloomFilter::myBloomFilter(int mm){
    //predetermine all hash lengths
    this->hash_length = 4;
    //length of filter
    this->length = mm;
    //computes number of hashes with minimal probability of false positive
    this->kk = ceil(log(2)*((this->length)/(this->hash_length)));
    
    //initializes an empty myBloomFilter
    for(int ii = 0; ii < this->length; ii++)
    {
        storage.push_back(0);
    }
    //cout << "this length is " << this->length<< endl;
}

myBloomFilter::myBloomFilter(string filter){
    //predetermine all hash lengths
    this->hash_length = 4;
    //length of filter
    this->length = filter.length();
    //computes number of hashes with minimal probability of false positive
    this->kk = ceil(log(2)*((this->length)/(this->hash_length)));
    bool value;
    
    //initializes bloom filter to specific input string
    for(int ii = 0; ii < this->length; ii++){
        value = ((filter[ii] == '1') ? true : false);
        this->storage.push_back(value);
    }
    //cout << "this length is " << this->length<< endl;
}

//goes through a series of hashes and then turns the specific bits on
void myBloomFilter::insert(string item){
    int bloomIndex;
    int runsum = 0;
    string hash_string;
    
    //runs each hash with variation ii
    for(int ii = 1; ii <= this->kk; ii++){
        hash_string = hash(item,ii);
        //for debugging purposes
        //cout << "hash "<< ii << " for " << item  <<  " returned a hash of "<< hash_string << endl;
        //cout << hash_string << endl;
        
        for(int jj = 0; jj < hash_length; jj++){
            runsum += hash_string[jj] + 0;
        }
        //each hash will produce a special index in the myBloomFilter
        bloomIndex = runsum % this->length;
        //flips the appropriate bits to account for item insertion
        if(storage.at(bloomIndex) == false)
            storage.at(bloomIndex) = true;
    }
}

//checks if the item (once hashed) is probably in the rainbow table
bool myBloomFilter::exists(string item){
    bool result = true;
    int bloomIndex;
    int runsum = 0;
    string hash_string;
    
    for(int ii = 1; ii <= this->kk; ii++){
        hash_string = hash(item,ii);
        for(int jj = 0; jj < hash_string.length(); jj++){
            runsum += hash_string[jj] + 0;
        }
        bloomIndex = runsum % this->length;
        if(storage.at(bloomIndex) == false){

<p><font face="verdana"><b>Explaination: </b>
My code allows initialization of a bloom filter by either creating a filter of size mm or by passing a pre-made bloom filter in the form of a string. The length of the filter is immediately saved in a member variable. This will be very important for later. A  vector (of booleans to conserve space) is being used to handle the back-end storage of the bloom filter.</font></p>

hash:

This function takes in strings and then converts them to hashes-- essentially indecipherable jumble. These hashes are then converted to specific indices used to turn on the bits in the filter (the conversion occurs in the insert function). My particular hash function is dependents on a second argument that manipulates the behavior of the hashes. This hash is a slight variation of the popular division method hash.

insert:

Simply runs the hash functions for the given input item, converts the hashes to indices and then turns on the bits of these indices if the specific bit in the filter is off (otherwise the bit is already on), which despite potentially creating collisions, the unique hashes will ensure no false negatives, and minimal false positives. 

exists:

Runs all the hash functions for the given input items, converts the hashes to indices, and then checks all corresponding indices that are returned. If the converted index holds a false for any of the converted hashes, the item is definitely not a member.

output:

Translates the back-end storage of booleans into an output-able string
